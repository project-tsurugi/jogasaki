# NULL比較セマンティクスの修正

2026-04-28 kurosawa

## 本文書について

SQL実行エンジンのNULL比較セマンティクスの修正方針についてまとめる

## 用語

- 等価比較: 通常のスカラ値の比較演算子 `=` を使用するセマンティクス。比較結果は3値論理となり、NULLを含む比較は unknown とする (a.k.a null aware comparison)

- 非区別比較: `IS NOT DISTINCT FROM` による比較。比較結果は2値論理(true/false)。表記が必要な場合は null-safe operator `<=>` で表す。
NULLどうしは等しい (i.e. `NULL <=> NULL` が true) ものとして比較される。(a.k.a null unaware comparison)

## 現状の問題点

詳細はドキュメント [NULL比較セマンティクスの問題点](./nulls-comparison.md) を参照のこと。要約すると、以下のような問題がある。

1. シャッフルベースでの結合処理が非区別比較で実装されていて、NULLを結合キーにした場合にSQL標準と異なる結果になる
2. IN演算子でサブクエリを右辺にとるケースも(semi-join処理によって実装されており) 非区別比較となり正しくない (issues #1486) 

これとは別に、調査中に発見された以下の問題があるがこれについては別途対応する

- 範囲演算検索の処理において、NULL及び+Inf/-Inf の端点処理に誤りがありセカンダリインデックスを使用した際に不正なクエリ結果となる (issue #1484, #1479)

## 修正案

### 方針

- シャッフル結合処理(inner/left outer/full outer/anti/semi join)について、要求セマンティクスが「等価比較」か「非区別比較」かによって正しい比較を行うように修正し、クエリ全体としてはSQL標準に従うようにする
  - 具体的にはjoin演算子(`takatori::relation::step::join`)を拡張し、等号による結合条件部分が等価比較か非区別比較をコンパイラから受け取れるようにする。例えばコンパイラは下記のようにクエリの処理によってこれらの比較セマンティクスを切り替える。
    - IN-subquery 演算子は semi-join + 等価比較 
    - INTERSECT DISTINCTは semi-join + 非区別比較
    - EXCEPT DISTINCTは anti-join + 非区別比較
  - 例外として、 anti-join + 等価比較 はサポート外とする
    - シャッフル結合処理による対応が難しいため
    - これができると NOT IN-subquery が実現可能な見込みだが、現時点では引き続き未サポートの予定
  - スカラ式による等号比較はこれまで通り等価比較を行う(`takatori::relation::step::join::condition()` 部分を含む)
  - group/aggregate エクスチェンジや `take_cogroup` 演算子についてはこれまで通り非区別比較を行う
    - 将来的な拡張用に比較セマンティクスを渡せるようにしてもよいが、現状の機能の範囲では非区別比較のみで十分である見込み

### 動作原理と実装イメージ

下記でこの修正案の詳細と動作の正しさについて説明する

#### シャッフル結合処理の流れ 

シャッフル結合処理における典型的な処理内容と、どのような比較演算が行われるかを下記に示す。

- groupエクスチェンジがグルーピングキーに基づいてレコードをグループ化して、グループメンバをまとめて下流に渡す
  - メンバー同士をグループ化するにあたってグルーピングキーの比較を行う
- take_cogroup 演算子は複数の groupエクスチェンジからグループを受取り、グルーピングキーが等しいもの同士を結合させて cogroupレコードを作成する
  - 複数のグループを結合させるにあたってグルーピングキーの比較を行う
- join 演算子が take_cogroup 演算子からcogroupレコードを受取り、結合種別(innerやfull等)ごとに分岐してグルーピングキー以外の結合条件を評価し、結合に成功したレコードを(フラットなレコードとして)下流に渡す
  - グルーピングキー以外の結合条件に等号・不等号が含まれる場合はその比較を行う
  - グルーピングキーについての比較はすでに上流で済んでいるため join演算子内では行わない

#### 修正詳細

1. これまで通りgroupエクスチェンジやtake_cogroup演算子は非区別比較を行う。つまり、join演算子が受け取るcogroupレコードは各エクスチェンジからのグループでグルーピングキーが非区別比較の意味で等しいものからなる。

2. join演算子が上流から渡された cogroup レコードの評価を行う際、要求される比較セマンティクスとjoin種別(inner等)によって NULLをグルーピングキーに含む cogroupの取り扱いを分岐する。

具体的には、既存の結合処理ロジックでメンバー同士が「マッチする」という条件評価を行う際に下記a,bの条件によって行っていたが、ここにcの条件を追加する。

    a. メンバが同一のcogroupレコードに属すること(つまりグルーピングキーが等しい)
    b. 結合条件の評価結果がtrueである
    c. (**グルーピングキーのNULL条件**) 以下のいずれかを満たすこと
      - join演算子に要求される比較セマンティクスが等価比較であって、グルーピングキーのすべてが非NULLであること
      - join演算子に要求される比較セマンティクスが非区別比較であること

条件 a により、マッチ成功または失敗の判定はcogroupレコード単体で完結し、異なるcogroupレコードのメンバを参照する必要はない。

条件 c によって、NULLを含むグルーピングキーの比較が正しくマッチの成否に反映される。

### 結合種別ごとの処理内容

join演算子はcogroupレコード内のメンバ同士のマッチ判定を行い、その成否と結合種別を使って結果レコード作成する。下記では結合種別ごとの処理について説明する。

下記の記法を用いて、
`(k1, k2, ..., G1, G2, ...)` という形で表されるcogroup レコードが 
どのように処理されるかを説明する。

- `k1`, `k2`, ... : グルーピングキーの値
- `G1`, `G2`, ... : グループ (=メンバーの集合)
- `cond` : join演算子に要求される等号条件以外の結合条件(extra condition)
- `eq_sem` : join演算子に要求される比較セマンティクス。等価比較(`=`)か非区別比較(`<=>`)のいずれか


グループ `G` のメンバの列をすべて NULL としたものを `G` の NULLメンバと呼ぶ。

##### inner join 処理 

- `eq_sem` が `=` かつ k1, k2, ... のうち少なくとも1個がNULLの場合、全メンバについてマッチは失敗。この cogroup レコードの処理結果は空とする
- 上記以外の場合、G1, G2, ... のメンバーどうしの全ての組合せについて、`cond` を `eq_sem` のもとで評価して `true` であればマッチ成功とし、結合したレコードを下流へ渡す

##### left outer join

- `eq_sem` が `=` かつ k1, k2, ... のうち少なくとも1個がNULLの場合、全メンバのマッチは失敗だが、最左端のメンバを残すために G1のメンバと G2, G3, ... のNULLメンバを結合させ、この cogroup レコードの処理結果とする
- 上記以外の場合、G1のメンバーと G2, G3, ... のメンバーどうしの全ての組合せについて、`cond` を `eq_sem` のもとで評価して `true` であればG1のメンバはマッチ成功、結合したレコードを下流へ渡す
  - 1度もマッチしなかったG1のメンバがある場合、それと G2, G3, ... のNULLメンバを結合させたレコードを下流へ渡す

##### full outer join

full outer joinは2項に限定されているため、グループはG1, G2の2つとする

- `eq_sem` が `=` かつ k1, k2, ... のうち少なくとも1個がNULLの場合、G1のメンバと G2のNULLメンバ、およびG1のNULLメンバとG2のメンバを結合させたものをこの cogroup レコードの処理結果とする
- 上記以外の場合、G1のメンバーとG2のメンバーどうしの全ての組合せについて、`cond` を `eq_sem` のもとで評価して `true` なものがあればそのメンバどうしはマッチ成功、結合したレコードを下流へ渡す
  - 1度もマッチしなかったG1/G2のメンバがある場合、それと G2/G1のNULLメンバを結合させたレコードを下流へ渡す

##### semi join

- `eq_sem` が `=` かつ k1, k2, ... のうち少なくとも1個がNULLの場合、この cogroup レコードの処理結果は空とする
- 上記以外の場合、G1のメンバーと G2, G3, ... のメンバーどうしの全ての組合せについて、`cond` を `eq_sem` のもとで評価して `true` なものが1つ以上あればそのG1のメンバはマッチ成功、G1のメンバからなるレコードを下流へ渡す

##### anti join

方針セクションで述べた通り、 `eq_sem` が `<=>` の場合のみサポートとする

- G1のメンバーと G2, G3, ... のメンバーどうしの全ての組合せについて、`cond` を `eq_sem` のもとで評価し、`true` なものが 0個 であればそのG1のメンバはマッチ失敗、G1のメンバからなるレコードを下流へ渡す

## その他の修正案

上記は修正案(松)と呼んでいたもの。
下記の修正案も検討したが、完全な修正にならないのと作業量も少なくないため廃案とした

## 修正案(竹)

- `=` セマンティクスを使用しているところを `<=>` セマンティクスに修正する

    - スカラ式の等号による評価 : `=`
    - インデックスジョイン(join_find/join_scan) : `=` 
    - find/scan : `=` 
    - IN演算子で右辺が値のリストの場合 `=` 

### 見積もり

jogasaki: 7-9d

既存コードの多くの場所で `=` セマンティクスを採用している箇所が多いため、変更の影響範囲が大きい(演算子4種類、および式評価器)

## 修正案(梅)

- 既存のセマンティクスが混在している状況をそのままにする

- シャッフルによる結合処理 がSQL非標準になるが、GA時からずっとそうなので、一旦そのままとする
- 右辺がサブクエリの場合 `<=>` (**semi join使用によるバグ: issue #1486**) についてもそのままにするか修正するかは要検討

### 見積もり

1d (#1486 を除く)

影響範囲の確認と合意。エラーを出す場合は +2d 程度
