# アサートに関する方針

2026-04-09 kurosawa

## 本文書について

jogasakiのコードにおいて関数の不変条件や変数の契約を確認するためのアサートをどう使うかの方針についてまとめる

## 背景

- jogasakiはリリースビルド (`RelWithDebInfo`) でも `NDEBUG` を定義しない
- 以前はアサートのために `BOOST_ASSERT` を使用していたが、リリースビルドでアサートが発現し Tsurugiサーバーがクラッシュすることがあった
- これを避けるため、 `jogasaki::utils::assert_with_exception` を用意して、アサート条件に失敗した場合は例外を投げるようにした
- しかし `noexcept` 修飾された関数内で例外を投げると `std::terminate` によりプログラムが終了するため、 `assert_with_exception` を使用する関数は `noexcept` であってはならない
- 一般的なコーディング慣習では関数を `noexcept` 修飾した方が性能がよくコードの意図も明確になり保守性が向上するが、 `assert_with_exception` とは両立できないため、`noexcept` 修飾をいつ行うかについてのポリシーが必要である

## 方針

- `noexcept` が強く求められる関数を除き、 `assert_with_exception` を使用してアサートを行ってよい
  - すでに関数に `noexcept` の修飾があるものは外す必要がある
  - その関数を呼び出す側も同様に `noexcept` を外すこと
- `noexcept` が強く求められる関数は下記の通り
    - クラスのデストラクタ
      - 例外発生時にスコープ脱出にともないスタック上のオブジェクトに対するデストラクタが呼び出される
        - デストラクタが例外を投げると、例外の伝播中にさらに例外が発生することになり、 `std::terminate` によりプログラムが終了する
      - この危険性のため、デストラクタは暗黙的に `noexcept` であり、例外を投げるデストラクタが許されるケースはほとんどない
    - クラスのムーブコンストラクタとムーブ代入演算子
      - `std::vector` 等の標準コンテナのリアロケーション時、要素のムーブ操作が例外を投げる場合は例外安全性のためにコピー操作にフォールバックする
    - `swap` 関数
      - 例外安全性のため、`swap` 関数は例外を投げないことが推奨される
