# データ入り既存テーブルへの CREATE INDEX の設計

2026-04-14 kurosawa

## 本文書について

テーブルにデータがすでに存在している場合の CREATE INDEX 文の実行を可能にする機能の設計を記述する。

## 背景・問題

- jogasaki の現行実装では、CREATE INDEX 実行時にベーステーブルのストレージがすでにデータを持っている場合、インデックスの構築処理は行われずエラーを返す
- 実運用においては、テーブルにデータを投入した後にインデックスを追加したいユースケースが頻繁に発生する
- 本機能はこの制限を解消し、任意のタイミングでのインデックス追加を可能にすることを目的とする

## 実装方針

### フルスキャンによるインデックス構築

CREATE INDEX DDL の実行時に、ベーステーブルの全行をスキャンして対象のインデックスストレージへ挿入する。

- shirakami でインデックスのストレージを作成した後、ベーステーブルをフルスキャンして各行からインデックスキーを生成し、インデックスストレージに書き込む
- スキャン・書き込みが完了してからインデックスのメタデータを更新し(完成フラグをセット)し、以降の DML がインデックスを利用できるようにする

### DDL 処理のタスク化

インデックス構築にはフルスキャンを含むため、応答時間が長くなる場合がある。これに対応するため、DDL 処理をリクエストスレッドで同期的に実行するのではなく、タスクとしてスケジュールする方式へ移行する。

- CREATE INDEX のリクエストを受け付けたリクエストスレッドは、インデックス構築タスクをスケジュールして即座に戻る
- インデックス構築タスクはスケジューラが管理するスレッド上で非同期に実行される
- クライアントはタスクの完了通知を受け取ることで DDL の完了を知る

### スキャンの複数トランザクション分割

テーブル閉塞（後述）により CREATE INDEX 実行中はテーブルへの更新が禁止されるため、スキャンを単一トランザクションで完結させる必要はない。複数のトランザクションに分割してスキャンを行う。

- スキャンを複数のトランザクションに分割することで、shirakami のリードセットが巨大化することを防ぐ
- 各トランザクションでテーブルの一部の行をスキャンし、対応するインデックスエントリを書き込んだ後、コミットする
- 次のトランザクションでその続きの行をスキャンする
- テーブル閉塞によりスキャン中に更新が発生しない保証があるため、この分割方式でもインデックスの内容に矛盾は生じない

>[!Note]
> ベーステーブルの読取りにRTXを使用する案もあったが下記の理由により単純にOCCを利用する方式とする
> - テーブル閉塞によりRTXでなくてもスキャンの分割が可能
> - RTXを使用した場合は CREATE INDEX の実行開始タイミングでの最新データとRTXが読む safe snapshot が一致している確認が必要になる
> - RTXを使用すると read set を作成しないためメモリ消費量の観点では有利だが、OCCでも分割によりスキャン範囲を小さくできるため、メモリ消費量をコントロールできる

### クラッシュ時の扱い

CREATE INDEX の実行中にプロセスがクラッシュした場合、以下の仕様とする。

- shirakami のストレージはペンディング（未完成）状態で残留する
  - CREATE INDEX の開始時にストレージメタデータにはペンディング状態を示すフラグを設定する
  - CREATE INDEX の完了時にフラグをクリアする(クライアントにも完了通知を送る)
- jogasaki 起動時のメンテナンス処理またはメンテナンススレッドが未完成のストレージを検出し、削除する
- クラッシュ時、クライアントは CREATE INDEX の完了を受け取っていないため、再度 CREATE INDEX を発行する必要がある

>[!Note]
> ストレージメタデータは shirakami のストレージにストレージオプションとして記録している。上記のロジックは shirakami のストレージオプションの更新が素早く完了して永続化されることを前提としている。ただし実際は永続化までに時間がかかる可能性があるため、CREATE INDEX の完了通知からストレージメタデータの永続化までの間にクラッシュした場合は、ペンディング状態のストレージが残る可能性がある。
> ストレージオプションの更新をDDLを実行しているトランザクション上で行い、そのトランザクションのコミットと永続化完了通知を待ってから CREATE INDEX の完了通知をクライアントに送るといった対応が必要になるが、DDL とそれを実行するトランザクションのコミットについては詳細な方針が未確定のため、初動ではこの問題には対処しない。

## テーブル閉塞

CREATE INDEX の実行中に対象テーブルのデータが更新されると、作成中のインデックスと実テーブルの間に不整合が生じる。これを防ぐためのテーブル閉塞について、2 つの方式を検討する。

共通の前提として、作成中のインデックスストレージには CREATE INDEX を実行する DDL トランザクションが排他ロックを取得する。

---

### ALTER 方式

ALTER TABLE 等の DDL と同様に、ベーステーブルに対しても排他ロックを取得する方式。

#### 動作

- CREATE INDEX DDL 開始時に、作成中のインデックスに加えてベーステーブルにも排他ロックを取得する
- 排他ロックが取得されると、以降に届く当該テーブルを操作対象とするリクエストはすべてエラーを返す
- DDL 開始時点で、当該テーブルを操作対象とする未コミットトランザクション（DML）はアボートさせる
- 同一テーブルに対する複数の CREATE INDEX を同時に実行しようとした場合、先行の CREATE INDEX が排他ロックを保持しているため、後続はエラーになる（同時に 1 件のみ実行可能）

#### 利点・欠点

- 利点：実装が比較的シンプル
- 欠点：CREATE INDEX の実行期間中、当該テーブルへの DML がすべて禁止されるため可用性が低い

---

### SHARED 方式

排他/共有の 2 段階ではなく、共有ロックを readers/writers/others の 3 種類に細分化することで、CREATE INDEX と読み取り専用 DML の並列実行を可能にする方式。

#### ロックの種類

| ロック種別 | 意味 | 取得主体 |
|---|---|---|
| exclusive | テーブルへの排他アクセス | DDL（CREATE/DROP TABLE など） |
| shared + writers | テーブルへの共有アクセス（更新あり） | IUD（INSERT/UPDATE/DELETE）を含む DML |
| shared + readers | テーブルへの共有アクセス（読み取り専用スキャン） | CREATE INDEX が発行するスキャン |
| shared + others | テーブルへの共有アクセス（その他） | 読み取り専用クエリ（SELECT） |

#### ロックの競合規則

- exclusive は他のすべてと競合する
- shared 同士は exclusive と競合しない
- writers と writers は競合しない（複数の更新 DML は並列実行可能）
- readers と readers は競合しない（複数の CREATE INDEX スキャンは並列実行可能）
- readers と writers は競合する（CREATE INDEX スキャン中は IUD が禁止される）
- others は readers・writers のいずれとも競合しない

#### 動作

- CREATE INDEX DDL 開始時に、作成中のインデックスへの排他ロックに加えて、ベーステーブルに shared + readers ロックを取得する
- shared + readers が保持されている間は、ベーステーブルへの shared + writers ロックを必要とする IUD を含む DML を禁止する
  - 純粋な読み取り専用クエリ（shared + others）は CREATE INDEX と競合せず、並列実行可能
- 同一テーブルに対する複数の CREATE INDEX は互いに readers ロックを取得するため、競合せず並列実行可能
- IUD を含む DML はスキャン完了後にのみ再び受け付けられるようになる

#### 利点・欠点

- 利点：読み取り専用クエリを CREATE INDEX に対してブロックしない。同一テーブルへの複数 CREATE INDEX を並列化可能
- 欠点：ロックの種類が増えるため実装が複雑になる

---

## その他・要調査事項

- スキャン分割の境界（どの粒度で 1 つのトランザクションとするか）の具体的な値
- SHARED 方式における readers/writers 競合の排他実装（既存ロックマネージャの拡張方針）
- ALTER 方式の「すでに実行中の DML をアボートさせる」実装の具体的な方法
- CREATE INDEX が失敗した場合（エラー終了）のストレージクリーンアップ
- 複数インデックスを持つテーブルへの DROP INDEX と CREATE INDEX が同時に発行された場合の競合処理
