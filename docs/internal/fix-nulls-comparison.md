# NULL比較セマンティクスの修正

2026-04-28 kurosawa

## 本文書について

SQL実行エンジンのNULL比較セマンティクスの修正方針についてまとめる

## 現状の問題点

詳細はドキュメント [NULL比較セマンティクスの問題点](./nulls-comparison.md) を参照のこと。要約すると、以下のような問題がある。

1. シャッフルベースでの結合処理が `=?` セマンティクスで実装されていて、NULLを結合キーにした場合にSQL標準と異なる結果になる
2. IN演算子でサブクエリを右辺にとるケースも(semi-join処理によって実装されており) `=?` セマンティスクスで正しくない (issues #1486) 

これとは別に、調査中に発見された以下の問題があるがこれについては別途対応する

- 範囲演算検索の処理において、NULL及び+Inf/-Inf の端点処理に誤りがありセカンダリインデックスを使用した際に不正なクエリ結果となる (issue #1484, #1479)

## 修正案(松)

- anti/semi joinも含め、シャッフルベースでの結合処理を null awareかunawareかに応じて正しく修正し、SQL標準に従うようにする
  - コード変更
    - join演算子
      - 等号による結合条件部分について、null awareかunawareかをコンパイラから受け取れるようにする 
        - null aware: 等号による結合条件をもつ内部・外部結合、IN-subquery 演算子(semi-join)
        - null unaware: 上記以外のケース、特に INTERSECT DISTINCT (semi-join), EXCEPT DISTINCT (anti-join) を含む
      - null awareかどうかによって、NULLをcogroupのキー列に含むものの処理を分岐する
        - null awareの場合、結合失敗として処理する。joinの種類によって以下のようになる
          - inner join: cogroupは捨てて無視する
          - left outer join: cogroup内の左側のレコードを保存する
          - full outer join: cogroup内の両側のレコードを保存する
          - semi join: cogroupは捨てて無視する
          - anti join: cogroup内の左側のレコードを保存する
        - null unawareの場合、結合成功として処理する。キーにNULLを含むだけで、cogroupの処理は non-NULLなキーと同様に行う
      - 等号以外の結合条件部分はこれまで通り null aware : e.g. NULL < 1 などは unkonwn

  - 変更不要 
    - group/aggregate エクスチェンジ
      - グループ化キーにNULLが含まれていても、`=?` を使用して処理。下流へグループとして渡す。 
    - take_cogroup 演算子
      - cogroupのキーにNULLが含まれていても、`=?` を使用して処理する

### 見積もり

jogasaki: 4-5d 

修正が join 演算子に集中しており、下記修正案(竹)に比べてjogasaki側の変更は大きくない

## 修正案(竹)

- `==` セマンティクスを使用しているところを `=?` セマンティクスに修正する

    - スカラ式の等号による評価 : `==`
    - インデックスジョイン(join_find/join_scan) : `==` 
    - find/scan : `==` 
    - IN演算子で右辺が値のリストの場合 `==` 

### 見積もり

jogasaki: 7-9d

既存コードの多くの場所で `==` セマンティクスを採用している箇所が多いため、変更の影響範囲が大きい(演算子4種類、および式評価器)

## 修正案(梅)

- 既存のセマンティクスが混在している状況をそのままにする

- シャッフルによる結合処理 がSQL非標準になるが、GA時からずっとそうなので、一旦そのままとする
- 右辺がサブクエリの場合 `=?` (**semi join使用によるバグ: issue #1486**) についてもそのままにするか修正するかは要検討

### 見積もり

1d (#1486 を除く)

影響範囲の確認と合意。エラーを出す場合は +2d 程度
