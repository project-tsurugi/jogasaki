# NULL比較セマンティクスの修正

2026-04-28 kurosawa

## 本文書について

SQL実行エンジンのNULL比較セマンティクスの修正方針についてまとめる

## 用語

- 等価比較: 通常のスカラ値の比較演算子 `=` を使用するセマンティクス。比較結果は3値論理となり、NULLを含む比較は unknown とする (a.k.a null aware comparison)

- 非区別比較: `IS NOT DISTINCT FROM` による比較。比較結果は2値論理(true/false)。表記が必要な場合は null-safe operator `<=>` で表す。
NULLどうしは等しい (i.e. `NULL <=> NULL` が true) ものとして比較される。(a.k.a null unaware comparison)

## 現状の問題点

詳細はドキュメント [NULL比較セマンティクスの問題点](./nulls-comparison.md) を参照のこと。要約すると、以下のような問題がある。

1. シャッフルベースでの結合処理が非区別比較で実装されていて、NULLを結合キーにした場合にSQL標準と異なる結果になる
2. IN演算子でサブクエリを右辺にとるケースも(semi-join処理によって実装されており) 非区別比較となり正しくない (issues #1486) 

## 修正案

### 方針

- シャッフル結合処理(inner/left outer/full outer/anti/semi join)について、要求セマンティクスが「等価比較」か「非区別比較」かによって正しい比較を行うように修正し、クエリ全体としてはSQL標準に従うようにする
  - 具体的にはjoin演算子におけるスカラ式 (`takatori::relation::step::join::condition()`) の評価を拡張し、**等号による結合条件部分が等価比較か非区別比較か**をコンパイラから受け取れるようにする。コンパイラは条件式を適切に渡すことで、shuffle join全体として下記のような比較セマンティクスを実現する。
    - IN-subquery は semi-join + 等価比較 
    - INTERSECT DISTINCTは semi-join + 非区別比較
    - EXCEPT DISTINCTは anti-join + 非区別比較
    - 相関サブクエリは left-outer-join + 非区別比較 (サブクエリが参照するパラメータ集合を再度メインクエリの結果と合流させる際の結合 [参照](https://nautilus-technologies.app.box.com/file/2139992855044))
  - 例外として、 anti-join + 等価比較 はサポート外とする
    - シャッフル結合処理による対応が難しいため
    - これができると NOT IN-subquery が実現可能な見込みだが、現時点では引き続き未サポートの予定

  - group/aggregate エクスチェンジや `take_cogroup` 演算子についてはこれまで通り非区別比較を行う
    - 将来的な拡張用に比較セマンティクスを渡せるようにしてもよいが、現状の機能の範囲では非区別比較のみで十分である見込み

### 動作原理と実装イメージ

下記でこの修正案の詳細と動作の正しさについて説明する

#### シャッフル結合処理の流れ 

シャッフル結合処理における典型的な処理内容と、どのような比較演算が行われるかを下記に示す。

- groupエクスチェンジがグルーピングキーに基づいてレコードをグループ化して、グループメンバをまとめて下流に渡す
  - メンバー同士をグループ化するにあたってグルーピングキーの比較を行う
- take_cogroup 演算子は複数の groupエクスチェンジからグループを受取り、グルーピングキーが等しいもの同士を結合させて cogroupレコードを作成する
  - 複数のグループを結合させるにあたってグルーピングキーの比較を行う
- join 演算子が take_cogroup 演算子からcogroupレコードを受取り、結合種別(innerやfull等)ごとに分岐してグルーピングキー以外の結合条件を評価し、結合に成功したレコードを(フラットなレコードとして)下流に渡す
  - グルーピングキー以外の結合条件に等号・不等号が含まれる場合はその比較を行う
  - グルーピングキーについての比較はすでに上流で済んでいるため join演算子内では行わない

#### 修正詳細

1. これまで通りgroupエクスチェンジやtake_cogroup演算子は非区別比較を行う。つまり、join演算子が受け取るcogroupレコードは各エクスチェンジからのグループでグルーピングキーが非区別比較の意味で等しいものからなる。

2. SQLコンパイラは join演算子全体として実現すべき比較セマンティクスとjoin種別(inner等)を考慮し、NULLをグルーピングキーに含むレコードを適切に処理する結合条件を追加する。

具体的には、既存の結合処理ロジックでメンバー同士が「マッチする」という条件評価を行う際に下記a,bの条件によって行っていたが、ここにcの条件を追加する。

    a. メンバが同一のcogroupレコードに属すること(つまりグルーピングキーが等しい)
    b. 結合条件の評価結果がtrueである
    c. (**グルーピングキーのNULL条件**) 以下のいずれかを満たすこと
      - join演算子に要求される比較セマンティクスが等価比較であって、グルーピングキーのすべてが非NULLであること
      - join演算子に要求される比較セマンティクスが非区別比較であること

条件 a により、マッチ成功または失敗の判定はcogroupレコード単体で完結し、異なるcogroupレコードのメンバを参照する必要はない。

条件 c によって、NULLを含むグルーピングキーの比較が正しくマッチの成否に反映される。
条件 c はコンパイラが注入する式によって実現される。

#### SQL コンパイラの表現

SQLコンパイラはスカラ式の等号比較で比較セマンティクスを受け取れるようにするため、`comparison_operator` に等価比較と非区別比較を表す値を追加する。

```
namespace takatori::scalar {

enum class comparison_operator {
    ...
    /// @brief is not distinct from
    is_not_distinct_from,
    /// @brief is distinct from (pseudo comparison operator for operator~())
    is_distinct_from,
    ...
};

...
```

#### jogasakiの修正内容

上述のように、本修正ではSQLコンパイラが適切な結合条件を設定して等価比較・非区別比較を指示する。

jogasakiは join演算子の結合条件(`takatori::relation::step::join::condition()`)の評価において、非区別比較を行う比較演算 (`comparison_operator` の `is_not_distinct_from` および `is_distinct_from`) をサポートする

- これまで `evaluator::engine::operator()(takatori::scalar::compare const& exp)` は NULL を含む比較に対しては unknown を返すようになっていたが、これを `comparison_operator` に応じて等価比較か非区別比較のいずれかを行うように修正する
  - `is_not_distinct_from` と `is_distinct_from` の評価の際は NULL を含む比較に対して true/false のいずれかを返す 
  - その他の `comparison_operator` についてはこれまで通り、NULL を含む比較に対しては unknown を返す

## その他の修正案

上記は修正案(松)と呼んでいたもの。
下記の修正案も検討したが、完全な修正にならないのと作業量も少なくないため廃案とした

## 修正案(竹)

- `=` セマンティクスを使用しているところを `<=>` セマンティクスに修正する

    - スカラ式の等号による評価 : `=`
    - インデックスジョイン(join_find/join_scan) : `=` 
    - find/scan : `=` 
    - IN演算子で右辺が値のリストの場合 `=` 

### 見積もり

jogasaki: 7-9d

既存コードの多くの場所で `=` セマンティクスを採用している箇所が多いため、変更の影響範囲が大きい(演算子4種類、および式評価器)

## 修正案(梅)

- 既存のセマンティクスが混在している状況をそのままにする

- シャッフルによる結合処理 がSQL非標準になるが、GA時からずっとそうなので、一旦そのままとする
- 右辺がサブクエリの場合 `<=>` (**semi join使用によるバグ: issue #1486**) についてもそのままにするか修正するかは要検討
